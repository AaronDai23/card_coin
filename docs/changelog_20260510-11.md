## 一、NFC 放卡自动跳转修复

### 问题
- **热启动**（app 已在前台）：放 NFC 卡时 Android OEM 弹窗或 app 内导航被触发
- **冷启动**（app 未运行）：NFC 启动 app 后 app_links 读取到 NFC intent，产生异常路由

### 文件
`android/app/src/main/kotlin/com/cardcoin/card_coin/MainActivity.kt`

### 改动
1. **热启动**：在 `onResume`/`onPause` 中恢复 `enableForegroundDispatch`/`disableForegroundDispatch`，将所有 NFC intent 路由到 `onNewIntent`，抑制 OEM 弹窗
2. **冷启动**：在 `configureFlutterEngine` 顶部检测 NFC launch intent，若命中则用 `ACTION_MAIN` 替换，阻止 app_links 读取到 NFC URL
3. **`onNewIntent`**：识别三种 NFC action（`NDEF_DISCOVERED`/`TECH_DISCOVERED`/`TAG_DISCOVERED`）后直接 return，不传给 super/app_links

## 三、提现 gasLimit 不足修复（EVM 链）

### 问题
部分 EVM 链（如 RTAP/RTBP）链上要求最低 gas > 21000，后端及 SDK 默认返回 21000 导致报错：
```
RPC error: intrinsic gas too low: gas 21000, minimum needed 21644
```

### 根因
`ChainClient.fetchEVMNativeFees()` 使用 `gasLimit = 21000` 计算 fee 估算，同时将 `"21000"` 作为 `FeeResponse.gasLimit` 返回给 Dart 层，最终传入链上签名。

### 改动的文件

#### 3.1 Android SDK（主要）
**`../../chipcore_sdk/android/app/src/main/kotlin/.../ChainClient.kt`**
```kotlin
// Before
val gasLimit = BigInteger.valueOf(21000)

// After
val gasLimit = BigInteger.valueOf(100000)
```
`FeeResponse` 中的 gasLimit 字符串也同步由 `"21000"` → `"100000"`（由变量自动生成）

**`../../chipcore_sdk/android/app/src/main/kotlin/.../ChipCoreBlockchainApi.kt`**
```kotlin
// native 转账默认 gasLimit（msg.gasLimit 为 null 时的兜底值）
// Before
txGasLimit = msg.gasLimit?.let { ... } ?: BigInteger.valueOf(21000)  // 实际已改为
// After
txGasLimit = msg.gasLimit?.let { ... } ?: BigInteger.valueOf(100000)
```

#### 3.2 iOS SDK
**`../../chipcore_sdk/ios/Runner/ChainClient.swift`**
```swift
// Before
let gasLimit: BigUInt = 21000
FeeResponse(..., gasLimit: "21000", ...)

// After
let gasLimit: BigUInt = 100000
FeeResponse(..., gasLimit: "100000", ...)
```

**`../../chipcore_sdk/ios/Runner/ChipCoreBlockchainApi.swift`**
```swift
// Before
txGasLimit = msg.gasLimit.flatMap { BigUInt($0) } ?? 21000

// After
txGasLimit = msg.gasLimit.flatMap { BigUInt($0) } ?? 100000
```

#### 3.3 Dart 层（双重保险）
**`lib/pages/main_page/hd_wallet_page/hd_send_page/effect.dart`**

EVM 链 native 转账在 Dart 层也强制覆盖为 100000，不依赖后端/SDK 返回值：
```dart
final isEvmChain = blockchainIdUpper.contains('ETH') ||
    blockchainIdUpper.contains('BSC') ||
    blockchainIdUpper.contains('BNB') ||
    blockchainIdUpper.contains('MATIC') ||
    blockchainIdUpper.contains('POLYGON') ||
    blockchainIdUpper.contains('RTAP') ||
    blockchainIdUpper.contains('RTBP');
String? safeGasLimit =
    (currencyType == 'blockchain' && isEvmChain) ? '100000' : gasLimit;
```

## 四、取消 PIN 后再次提现仍弹 PIN 框修复

### 问题
用户在 PIN 输入弹窗中点取消后，`state.lastPinSet` 仍为 `true`，导致下次发起提现时立即预弹 PIN 框，而不是先扫卡

### 文件
`../../chipcore_sdk/android/app/src/main/kotlin/.../ChipCoreBlockchainApi.kt`

### 改动
在 `sendTransaction` 和 `signDigest` 的预弹 PIN 弹窗取消路径中，重置 `state.lastPinSet = false`：

```kotlin
// sendTransaction
if (pinCode == null) {
    state.lastPinSet = false  // 新增：重置缓存
    mainHandler.post { result.error(flutterError("pin-cancelled", "PIN input cancelled")) }
    return@showPinInputDialog
}

// signDigest（同样修复）
if (pinCode == null) {
    state.lastPinSet = false  // 新增：重置缓存
    mainHandler.post { result.error(flutterError("pin-cancelled", "PIN input cancelled")) }
    return@showPinInputDialog
}
```

### 说明
- `handlePinRequired`（卡片当场返回 `pin-required` 错误）的取消路径**不重置** `lastPinSet`，因为那是卡片明确声明需要 PIN，下次扫卡仍需预弹
- 重置后，下次发送会先扫卡：若卡确实设有 PIN，`handlePinRequired` 会重新将 `lastPinSet = true` 并弹框；若卡已移除 PIN，则直接签名成功

## gasLimit 策略总结

| 场景 | gasLimit 来源 |
|---|---|
| EVM native 转账（ETH/BSC/MATIC/RTAP/RTBP） | 固定 `100000`（SDK + Dart 双层） |
| ERC20 token 转账 | `eth_estimateGas` 估算 × 1.2，兜底 `80000` |
| BTC / DOGE（UTXO 链） | 无 gasLimit 概念，SDK 忽略此字段 |
| TRX | 固定手续费，无 gasLimit |
